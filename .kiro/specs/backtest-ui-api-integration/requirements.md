# Requirements Document

## Introduction

ปัจจุบัน หน้า Backtest UI (`src/pages/Backtest.tsx`) รัน signal computation, trade simulation และ metrics calculation ทั้งหมดใน browser (client-side JS) โดยใช้ library `technicalindicators` และ logic ที่เขียนซ้ำกับ backend ทำให้เกิดปัญหาหลายอย่าง:

1. **Strategy ไม่ตรงกับ live bot** — UI มีแค่ 7 strategies แต่ SignalEngine มี 12 strategies (ขาด EMA_SCALP, STOCH_RSI, VWAP_SCALP, AI_SCOUTER)
2. **Logic ไม่ตรงกัน** — เช่น EMA_RSI ใน UI ใช้ threshold ต่างจาก backend
3. **ไม่รองรับ AI/Python strategies** — ไม่มี PYTHON: prefix support เลย
4. **Metrics ไม่ครบ** — ไม่มี Sharpe Ratio, Profit Factor, Avg Win/Loss, Max Consecutive Losses

ฟีเจอร์นี้จะ migrate UI ให้เรียก backend API endpoints ที่มีอยู่แล้ว (`POST /api/backtest/run`, `POST /api/backtest/compare`, `GET /api/backtest/history`, `GET /api/backtest/history/:backtestId`) แทนการคำนวณใน browser เพื่อให้ผลลัพธ์ backtest สอดคล้องกับ Binance live bot อย่างสมบูรณ์

---

## Glossary

- **Backtest_UI**: React component ที่ `src/pages/Backtest.tsx` — หน้า UI สำหรับรัน backtest
- **API_Gateway**: Backend service ที่ expose REST endpoints ที่ `/api/backtest/*`
- **BacktestResult**: ผลลัพธ์จาก `POST /api/backtest/run` ประกอบด้วย trades, metrics, equityCurve, backtestId
- **BacktestConfig**: ชุดพารามิเตอร์สำหรับรัน backtest (symbol, strategy, interval, tpPercent, slPercent, leverage, capital, startDate, endDate)
- **EquityCurve**: array ของ `{ time: string, value: number }` แสดงมูลค่า portfolio ตามเวลา
- **Trade**: ข้อมูล trade แต่ละรายการ ประกอบด้วย entryTime, exitTime, type, entryPrice, exitPrice, pnl, pnlPct, exitReason
- **BacktestSummary**: ข้อมูล backtest แบบย่อ (ไม่มี trades) ที่คืนจาก `GET /api/backtest/history`
- **CompareResult**: ผลลัพธ์จาก `POST /api/backtest/compare` ประกอบด้วย rank, configLabel และ BacktestResult fields
- **Leverage**: ตัวคูณ leverage สำหรับคำนวณ PnL (default 10)
- **SignalEngine**: โมดูล backend ที่คำนวณ signal — single source of truth สำหรับทุก strategy
- **JS Strategy**: strategy ที่ implement ใน SignalEngine (EMA, RSI, BB, GRID, EMA_RSI, BB_RSI, EMA_BB_RSI, AI_SCOUTER, EMA_SCALP, STOCH_RSI, VWAP_SCALP)
- **Python Strategy**: strategy ที่รันผ่าน strategy-ai service โดยใช้ prefix `PYTHON:` เช่น `PYTHON:bollinger_breakout`
- **Strategy Registry**: รายการ strategy ทั้งหมดที่ backend รองรับ ใช้เป็น source of truth สำหรับ UI dropdown

## Requirements

### Requirement 1: เรียก Backend API แทนการคำนวณใน Browser

**User Story:** As a trader, I want the backtest to run on the server using the same engine as the live bot, so that results are consistent and accurate.

#### Acceptance Criteria

1. WHEN the user clicks "Start Backtest", THE Backtest_UI SHALL send a `POST /api/backtest/run` request with the current BacktestConfig instead of computing signals and trades in the browser
2. THE Backtest_UI SHALL include all BacktestConfig fields in the request body: `symbol`, `strategy`, `interval`, `tpPercent`, `slPercent`, `leverage`, `capital`, `startDate` (if set), `endDate` (if set)
3. WHEN the API returns a BacktestResult, THE Backtest_UI SHALL display the result without performing any additional computation on the response data
4. IF the API returns an error response (4xx or 5xx), THEN THE Backtest_UI SHALL display the error message to the user and stop the loading state
5. THE Backtest_UI SHALL continue to call `GET /api/backtest` (klines endpoint) separately for the chart preview — this call is NOT replaced by the run endpoint

---

### Requirement 2: แสดง Metrics ที่ครบถ้วนจาก Backend

**User Story:** As a trader, I want to see comprehensive performance metrics from the server-side calculation, so that I can make informed decisions about strategy quality.

#### Acceptance Criteria

1. WHEN a BacktestResult is received, THE Backtest_UI SHALL display the following metrics from the response:
   - `totalPnl` and `netPnlPct` (Net PnL)
   - `winRate`, `totalTrades` (Win Rate / Trade count)
   - `maxDrawdown` (Max Drawdown %)
   - `profitFactor` (Profit Factor)
   - `sharpeRatio` (Sharpe Ratio)
   - `avgWin` and `avgLoss` (Avg Win / Avg Loss in USDT)
   - `maxConsecutiveLosses` (Max Consecutive Losses)
2. THE Backtest_UI SHALL remove the "Peak Float Loss" and "Open Positions" metrics that were computed client-side, as these are not returned by the backend API
3. WHEN `totalTrades` is 0, THE Backtest_UI SHALL display "0" or "--" for all metrics without errors

---

### Requirement 3: แสดง Equity Curve จาก Backend

**User Story:** As a trader, I want to see the equity curve generated by the server, so that I can visualize portfolio performance over time.

#### Acceptance Criteria

1. WHEN a BacktestResult is received, THE Backtest_UI SHALL render the `equityCurve` array from the response onto the Portfolio Equity Curve chart
2. THE equityCurve data format from the API is `Array<{ time: string, value: number }>` where `time` is an ISO 8601 string — THE Backtest_UI SHALL convert each `time` to a Unix timestamp (seconds) for the chart library
3. WHEN `equityCurve` is empty (no trades), THE Backtest_UI SHALL clear the equity chart without errors

---

### Requirement 4: แสดง Trade Markers บน Chart จาก Backend Trades

**User Story:** As a trader, I want to see entry and exit markers on the candlestick chart based on the server's trade results, so that I can visually inspect trade timing.

#### Acceptance Criteria

1. WHEN a BacktestResult is received, THE Backtest_UI SHALL generate chart markers from the `trades` array in the response
2. FOR each trade in the response, THE Backtest_UI SHALL place an entry marker at `entryTime` and an exit marker at `exitTime` on the candlestick chart
3. THE Backtest_UI SHALL color LONG entry markers green (below bar) and SHORT entry markers red (above bar), consistent with the current behavior
4. THE Backtest_UI SHALL label exit markers with the `exitReason` value from the trade ("TP", "SL", or "Signal Flipped")
5. WHEN `trades` is empty, THE Backtest_UI SHALL display no markers on the chart

---

### Requirement 5: แสดง Trade Log จาก Backend Trades

**User Story:** As a trader, I want to see the trade log populated from the server's trade data, so that I can review each trade's details.

#### Acceptance Criteria

1. WHEN a BacktestResult is received, THE Backtest_UI SHALL populate the Trade Log tab with the `trades` array from the response
2. THE Backtest_UI SHALL display the following fields for each trade: `entryTime`, `exitTime`, `type` (LONG/SHORT), `entryPrice`, `exitPrice`, `pnl`, `pnlPct`, `exitReason`
3. THE Backtest_UI SHALL display `pnlPct` as a percentage value alongside `pnl` in USDT in the trade log
4. THE Backtest_UI SHALL display trades in reverse chronological order (most recent first)
5. WHEN `trades` is empty, THE Backtest_UI SHALL display "Run backtest to see trade log." in the trade log table

---

### Requirement 6: เพิ่ม Leverage Input

**User Story:** As a trader, I want to configure leverage before running a backtest, so that PnL calculations reflect my actual trading setup.

#### Acceptance Criteria

1. THE Backtest_UI SHALL display a "Leverage" input field in the parameter panel with a default value of 10
2. WHEN the user changes the leverage value, THE Backtest_UI SHALL include the updated `leverage` value in the next `POST /api/backtest/run` request
3. THE Backtest_UI SHALL only show the Leverage input when the strategy is NOT "GRID" (Grid strategy does not use leverage)

---

### Requirement 7: History Panel — ดูผล Backtest ที่ผ่านมา

**User Story:** As a trader, I want to view a history of past backtest runs, so that I can compare results across different sessions.

#### Acceptance Criteria

1. THE Backtest_UI SHALL include a "History" tab alongside the existing "Charts" and "Trade Log" tabs
2. WHEN the user clicks the "History" tab, THE Backtest_UI SHALL call `GET /api/backtest/history` and display the returned list of BacktestSummary items
3. THE Backtest_UI SHALL display the following fields for each history item: `symbol`, `strategy`, `interval`, `totalPnl`, `winRate`, `totalTrades`, `createdAt`
4. WHEN the user clicks on a history item, THE Backtest_UI SHALL call `GET /api/backtest/history/:backtestId` and load the full BacktestResult into the UI (populating charts, metrics, and trade log)
5. IF the history list is empty, THEN THE Backtest_UI SHALL display "No backtest history yet." in the History tab
6. IF the `GET /api/backtest/history` call fails, THEN THE Backtest_UI SHALL display an error message in the History tab

---

### Requirement 8: Compare Mode — เปรียบเทียบหลาย Config

**User Story:** As a trader, I want to compare multiple strategy configurations side-by-side, so that I can identify the best parameters for a given symbol.

#### Acceptance Criteria

1. THE Backtest_UI SHALL include a "Compare Mode" toggle in the parameter panel
2. WHEN Compare Mode is active, THE Backtest_UI SHALL allow the user to add up to 10 BacktestConfig entries to a comparison list
3. WHEN the user clicks "Run Comparison", THE Backtest_UI SHALL send a `POST /api/backtest/compare` request with the `configs` array
4. WHEN CompareResults are returned, THE Backtest_UI SHALL display them in a ranked table showing: `rank`, `configLabel`, `totalPnl`, `winRate`, `sharpeRatio`, `maxDrawdown`, `profitFactor`
5. THE Backtest_UI SHALL sort the comparison table by `rank` ascending (rank 1 = best)
6. IF any config in the comparison result contains an `error` field, THEN THE Backtest_UI SHALL display that error inline in the table row for that config

---

### Requirement 9: Loading State และ Error Handling

**User Story:** As a trader, I want clear feedback while the backtest is running and when errors occur, so that I know the system is working and can act on failures.

#### Acceptance Criteria

1. WHILE a `POST /api/backtest/run` request is in-flight, THE Backtest_UI SHALL display a loading indicator and disable the "Start Backtest" button
2. WHEN the API returns an error with an `error` field in the response body, THE Backtest_UI SHALL display the error message to the user
3. IF the network request fails (no response), THEN THE Backtest_UI SHALL display "Network error — please check your connection" to the user
4. WHEN a new backtest run starts, THE Backtest_UI SHALL clear the previous results, trade log, markers, and equity curve before displaying new data

---

### Requirement 10: Strategy Dropdown ครบถ้วนตรงกับ SignalEngine

**User Story:** As a trader, I want the strategy dropdown to show all strategies available in the live bot, so that I can backtest any strategy I actually use.

#### Acceptance Criteria

1. THE Backtest_UI SHALL display all JS strategies registered in SignalEngine in the strategy dropdown:
   - `EMA` — EMA Crossover (20/50)
   - `RSI` — RSI Overbought/Oversold
   - `BB` — Bollinger Bands Mean Reversion
   - `EMA_RSI` — EMA + RSI Confirmation
   - `BB_RSI` — BB + RSI Confirmation
   - `EMA_BB_RSI` — Triple Confirmation (EMA + BB + RSI)
   - `GRID` — Grid Bot
   - `AI_SCOUTER` — AI Scouter
   - `EMA_SCALP` — EMA Scalp
   - `STOCH_RSI` — Stochastic RSI
   - `VWAP_SCALP` — VWAP Scalp
2. THE Backtest_UI SHALL NOT implement any signal computation logic in the browser — the strategy key is sent to the backend as-is
3. WHEN the user selects a strategy, THE Backtest_UI SHALL send the exact strategy key string (e.g. `"EMA_SCALP"`) to the API without transformation

---

### Requirement 11: Python Strategy Support

**User Story:** As a trader, I want to backtest Python-based AI strategies from the strategy-ai service, so that I can evaluate AI-powered strategies before deploying them live.

#### Acceptance Criteria

1. THE Backtest_UI SHALL include a "Python Strategy" mode toggle or section in the strategy selector
2. WHEN Python Strategy mode is active, THE Backtest_UI SHALL display a text input for the user to enter the Python strategy name (e.g. `bollinger_breakout`)
3. WHEN the user runs a backtest with a Python strategy, THE Backtest_UI SHALL prefix the strategy name with `PYTHON:` before sending to the API (e.g. `"PYTHON:bollinger_breakout"`)
4. IF the API returns `{ error: "Strategy AI service unavailable" }`, THEN THE Backtest_UI SHALL display "Strategy AI service is not available. Please ensure the strategy-ai service is running." to the user
5. THE Backtest_UI SHALL display a note near the Python strategy input explaining that results depend on the strategy-ai service being available
